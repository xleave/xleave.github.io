DROID 双机 Docker 部署中的环境兼容性与容器化工程问题

DROID 是一个面向 Franka Emika 机械臂的机器人数据采集与操控框架。在将其与 OpenPI 策略推理服务组合为双机（控制工作站 + 推理主机）Docker 部署方案时，容器化构建与运行阶段会暴露出一系列环境兼容性问题。

本文以问题驱动的形式，整理了部署过程中常见的 11 个工程问题及其解决方案。

Q1：克隆 DROID 仓库时子模块拉取失败，提示 Permission denied (publickey)

现象

Permission denied (publickey)
Could not read from remote repository

原因：DROID 上游仓库中部分子模块的远程地址采用 git@github.com: SSH 协议。在未配置 GitHub SSH Key 的干净构建环境（如 CI 或 Docker build）中，git submodule update 将因无法完成公钥认证而失败。

解决方案：通过 Git 一次性配置将 SSH 地址重写为 HTTPS，无需修改 .gitmodules：

git -c url.https://github.com/.insteadOf=git@github.com: \
    submodule update --init --recursive

Q2：Docker build 中 Conda 报 CondaToSNonInteractiveError

现象

CondaToSNonInteractiveError
Terms of Service have not been accepted

原因：较新版本的 Conda 在非交互环境中访问 repo.anaconda.com 频道前，要求显式接受 Anaconda 服务条款。docker build 是典型的非交互上下文。

解决方案：在 Dockerfile 中预先接受服务条款：

RUN conda tos accept --override-channels \
      --channel https://repo.anaconda.com/pkgs/main && \
    conda tos accept --override-channels \
      --channel https://repo.anaconda.com/pkgs/r

若使用的旧版 Conda 不支持 conda tos 子命令，可追加 || true 进行容错：

RUN conda tos accept --override-channels \
      --channel https://repo.anaconda.com/pkgs/main && \
    conda tos accept --override-channels \
      --channel https://repo.anaconda.com/pkgs/r || true

Q3：在 ubuntu:bionic 上运行最新版 Miniconda 安装器报 GLIBC 版本不足

现象

Installer requires GLIBC >=2.28, but system has 2.27.

原因：DROID 官方 NUC Dockerfile 基于 ubuntu:bionic（GLIBC 2.27）。当前 Miniconda3-latest-Linux-x86_64.sh 安装器要求 GLIBC ≥ 2.28。

临时方案：锁定一个仍兼容 GLIBC 2.27 的旧版安装器，例如：

https://repo.anaconda.com/miniconda/Miniconda3-py37_23.1.0-1-Linux-x86_64.sh

该方案可以绕过安装器的 GLIBC 检查，但后续 Conda 依赖求解的性能问题仍然存在。推荐最终方案见 Q4。

Q4：Conda 在求解 Polymetis 环境时长时间无响应

现象：执行 conda env create -f environment.yml 后，终端停留在 Solving environment: ...working... 数小时无进展。

原因：Polymetis 的 environment.yml 依赖图复杂且庞大，包含 Python、PyTorch CPU、Habitat、Pinocchio、hpp-fcl、Boost、gRPC、libfranka 等。经典 Conda SAT 求解器在此规模下性能极差。

解决方案：将包管理器从 Conda 替换为 Micromamba。Micromamba 使用 C++ 实现的 libsolv 求解器，可在数分钟内完成相同依赖图的求解：

ENV MAMBA_ROOT_PREFIX=/root/micromamba
ENV PATH=/usr/local/bin:$PATH

RUN wget -qO- https://micro.mamba.pm/api/micromamba/linux-64/latest | \
    tar -xvj -C /usr/local/bin --strip-components=1 bin/micromamba

RUN micromamba create -y -f ./polymetis/environment.yml && \
    micromamba run -n polymetis-local pip install -e ./polymetis && \
    micromamba clean -a -y

SHELL ["/usr/local/bin/micromamba", "run", "-n", "polymetis-local", \
       "/bin/bash", "-c"]

Q5：切换到 Micromamba 后，启动脚本报 conda.sh: No such file or directory

现象

/app/scripts/server/launch_server.sh: line 2:
  /root/miniconda3/etc/profile.d/conda.sh: No such file or directory

原因：上游 launch_server.sh 硬编码了 Conda 的激活路径（source conda.sh && conda activate）。将包管理器替换为 Micromamba 后，该路径不再存在。

解决方案：在 Dockerfile 构建阶段覆写启动脚本，将 Conda 激活替换为 Micromamba 激活：

RUN printf '%s\n' \
    '#!/bin/bash' \
    'set -e' \
    'cd /app/scripts/server' \
    'exec /usr/local/bin/micromamba run -n polymetis-local \
      python /app/scripts/server/run_server.py' \
    > /app/scripts/server/launch_server.sh

Q6：OpenPI Client 应该安装在哪个容器中？

背景：双机架构中，DROID Laptop 容器需要一个适配器（Adapter），将观测数据发送至 OpenPI Policy Server。OpenPI 官方仓库提供了 packages/openpi-client 和 examples/droid/main.py 作为参考实现。

推荐做法：

  • 将 openpi-client vendored 到 DROID 仓库（如 third_party/openpi-client）。
  • 编写适配器脚本（如 scripts/openpi/pi05_droid_policy_client.py）。
  • 仅在 Laptop Dockerfile 中安装 openpi-client。
  • NUC Dockerfile 不安装任何 OpenPI 依赖，保持控制侧镜像的最小化。

验证方式：

# NUC 镜像中不应存在 openpi_client
docker run --rm <nuc_image> python -c \
  'import importlib.util; print(importlib.util.find_spec("openpi_client"))'
# 期望输出: None

# Laptop 镜像中应当可以导入
docker run --rm <laptop_image> python -c \
  'import openpi_client; print(openpi_client.__file__)'

Q7：OpenPI Client 在 Python 3.7 环境中报语法或导入错误

现象：在 Python 3.7 环境中使用 OpenPI Client 时可能遇到类型注解语法错误、websockets API 不兼容、或额外依赖无法安装等问题。

原因及修改对照：

问题原始写法兼容修改
PEP 604 联合类型语法（Python ≥ 3.10）Dict[str, np.ndarray] | NoneOptional[Dict[str, np.ndarray]]
websockets 不同版本的参数名差异additional_headersextra_headers（配合 websockets==11.0.3）
tyro CLI 解析器引入额外依赖tyro替换为标准库 argparse

通过上述修改，适配器脚本可兼容 Python 3.7 环境且不引入额外依赖。

Q8：容器导入检查时报 ModuleNotFoundError 或 ImportError: libcuda.so.1

现象一

ModuleNotFoundError: No module named 'moviepy'

原因：DROID Laptop 的依赖安装在 Conda 环境（如 robot）中，直接调用系统 python3 使用的是系统 Python，不包含项目依赖。

解决方案：显式使用 Conda 环境中的 Python 解释器：

docker run --rm --entrypoint /bin/bash <laptop_image> \
  -lc '/root/miniconda3/envs/robot/bin/python -c \
    "import moviepy.editor; import openpi_client; print(\"ok\")"'

现象二

ImportError: libcuda.so.1: cannot open shared object file

原因：适配器的导入链涉及 ZED SDK 模块，后者依赖 NVIDIA 驱动库。在未挂载 GPU 的容器中这些库不可用。

解决方案：完整的导入验证需使用 NVIDIA Container Runtime：

docker run --rm --gpus all --entrypoint /bin/bash <laptop_image> \
  -lc '/root/miniconda3/envs/robot/bin/python -c \
    "import scripts.openpi.pi05_droid_policy_client as m; print(\"ok\")"'

Q9：docker ps 显示 OpenPI 容器 Up，但 Policy Server 实际未运行

现象：docker ps 显示容器状态为 Up，但客户端连接 Policy Server 端口时被拒。

原因：OpenPI 容器可能仅启动了交互式 Shell（bash），而非 serve_policy.py 进程。容器的 Up 状态仅表示主进程存活，不代表推理服务已就绪。

验证清单：

# 1. 检查容器内是否有 serve_policy.py 进程
docker top <openpi_container> -eo pid,comm,args

# 2. 检查端口是否已监听
nc -vz -w 2 127.0.0.1 8000

# 3. 检查宿主机网络接口是否已配置预期 IP
ip -br addr

三项检查均通过后，才可认为 Policy Server 处于可用状态。

Q10：使用 Host Network 模式时需要注意什么？

在 DROID 双机部署中，相关容器通常采用 network_mode: host。这一选择有以下影响：

  • 无 Docker NAT 端口映射：容器直接使用宿主机网络栈，端口冲突即为宿主机端口冲突。
  • Policy Server 端口直接绑定宿主机 IP：客户端应使用宿主机 IP 而非 172.17.0.x 等 Docker 桥接地址。
  • DROID/Polymetis 的 gRPC 端口（如 4242、50051、50052）同样直接暴露在宿主机上。
  • 跨主机启动不等于真实验证：在推理主机上启动 NUC 容器仅能验证镜像构建和依赖导入，不构成控制层面的真实测试，因为该主机通常缺少 RT 内核、Franka 网口和 Robotiq 串口等硬件资源。

Q11：PI05 DROID 策略使用几路相机？如何配置？

OpenPI 官方 pi05_droid 策略仅使用 两路视频流：

  • 1 路外部相机（External Camera）
  • 1 路腕部相机（Wrist Camera）

适配器中的推荐默认配置：

参数默认值说明
external_cameraleft指定使用左侧外部相机
right_camera_id空字符串可不配置，仅在需要双外部相机时填入

如果相机序列号尚未写入 droid/misc/parameters.py，则需要在运行命令中显式传入 --left-camera-id 和 --wrist-camera-id 参数。

附：Oculus/Quest 遥操作路径的裁剪

DROID 上游仓库包含 Oculus/Quest 遥操作代码。若部署目标为 OpenPI 策略自主控制而非头显遥操作，可从 Docker 构建路径中移除以下组件以精简镜像：

  • Oculus 相关的构建参数（如 NUC_OCULUS_DIR）
  • android-tools-fastboot、android-tools-adb 系统包
  • pip3 install -e oculus_reader
  • Compose 中 ADB 相关的环境变量

移除后应验证镜像中 oculus_reader 和 ppadb 模块均不可导入，adb、fastboot 命令均不存在。
