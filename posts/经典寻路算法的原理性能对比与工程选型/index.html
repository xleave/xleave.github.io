经典寻路算法的原理、性能对比与工程选型

寻路（Pathfinding）是图搜索理论在实际工程中最直接的应用之一。从游戏 NPC 的移动规划到机器人导航、地图路线推荐，寻路算法的选择直接影响系统的响应速度与路径质量。

本文系统梳理经典寻路算法的原理、性能特征与适用场景，帮助在工程实践中做出合理的选型决策。

1. 问题定义

寻路问题的形式化定义如下：给定一个图 G=(V,E)G = (V, E)G=(V,E)，其中 VVV 是节点集合，EEE 是边集合，每条边 e∈Ee \in Ee∈E 有非负权重 w(e)≥0w(e) \geq 0w(e)≥0。给定起点 sss 和终点 ttt，求从 sss 到 ttt 的最短路径（即路径边权之和最小的路径）。

根据图的表示方式，寻路场景可分为：

表示描述典型应用
网格图（Grid Graph）规则的二维或三维格子，每个格子为一个节点2D/3D 游戏地图、栅格地图
通用加权图节点和边的任意拓扑道路网络、社交网络
导航网格（NavMesh）由凸多边形构成的可行走区域3D 游戏引擎

2. 广度优先搜索（BFS）

2.1 原理

广度优先搜索从起点出发，逐层向外扩展，先访问距离为 1 的所有节点，再访问距离为 2 的所有节点，以此类推。它使用队列（FIFO Queue）维护待访问节点的顺序。

2.2 算法流程

function BFS(graph, start, goal):
    queue ← [start]
    visited ← {start}
    parent ← {start: null}

    while queue is not empty:
        current ← queue.dequeue()
        if current == goal:
            return reconstruct_path(parent, goal)
        for neighbor in graph.neighbors(current):
            if neighbor not in visited:
                visited.add(neighbor)
                parent[neighbor] ← current
                queue.enqueue(neighbor)

    return no path found

2.3 性能与适用场景

指标值
时间复杂度O(V+E)O(V + E)O(V+E)
空间复杂度O(V)O(V)O(V)
最优性仅在边权均等时保证最短路径
适用场景无权图或等权图的最短路径搜索

BFS 的核心限制在于它假设每条边的代价相等。当边权不同时（如不同地形的移动代价不同），BFS 无法保证找到最短路径。

3. Dijkstra 算法

3.1 原理

Dijkstra 算法是 BFS 在加权图上的推广。它维护一个从起点到每个已知节点的最短距离估计 d[v]d[v]d[v]，每次从未处理的节点中选择 ddd 值最小的节点进行扩展（贪心选择），并通过松弛（Relaxation）操作更新相邻节点的距离。

3.2 松弛操作

对于从节点 uuu 到 vvv 的边，松弛操作为：

if d[u]+w(u,v)<d[v]:d[v]←d[u]+w(u,v)\text{if } d[u] + w(u, v) < d[v]: \quad d[v] \leftarrow d[u] + w(u, v)if d[u]+w(u,v)<d[v]:d[v]←d[u]+w(u,v)

3.3 算法流程

function Dijkstra(graph, start, goal):
    dist ← {start: 0}      // 所有其他节点初始化为 ∞
    pq ← MinHeap([(0, start)])
    parent ← {start: null}

    while pq is not empty:
        (d, current) ← pq.extract_min()
        if current == goal:
            return reconstruct_path(parent, goal)
        if d > dist[current]:
            continue          // 跳过过期条目
        for (neighbor, weight) in graph.neighbors(current):
            new_dist ← dist[current] + weight
            if new_dist < dist.get(neighbor, ∞):
                dist[neighbor] ← new_dist
                parent[neighbor] ← current
                pq.insert((new_dist, neighbor))

    return no path found

3.4 性能与适用场景

指标值
时间复杂度O((V+E)log⁡V)O((V + E) \log V)O((V+E)logV)（二叉堆）
空间复杂度O(V)O(V)O(V)
最优性保证最短路径（要求非负边权）
适用场景非负权重图的单源最短路径

Dijkstra 的核心限制在于它是无方向性的搜索——它向所有方向均匀扩展，不利用终点位置信息。在大规模图中，这意味着大量不必要的节点被探索。

4. A* 算法

4.1 原理

A*（A-Star）算法是 Dijkstra 的有向扩展。它引入启发函数（Heuristic Function）h(n)h(n)h(n) 估计从节点 nnn 到终点的剩余代价，将节点的排序依据从 g(n)g(n)g(n)（起点到当前节点的实际代价）变为 f(n)=g(n)+h(n)f(n) = g(n) + h(n)f(n)=g(n)+h(n)。

这使得搜索优先探索”看起来更接近终点”的方向，显著减少不必要的节点扩展。

4.2 启发函数的要求

A* 保证最优性的前提是启发函数必须满足可采纳性（Admissibility）：

∀n:h(n)≤h∗(n)\forall n: \quad h(n) \leq h^*(n)∀n:h(n)≤h∗(n)

其中 h∗(n)h^*(n)h∗(n) 是从 nnn 到终点的真实最短距离。即启发函数不能高估真实代价。

常用的启发函数：

启发函数公式适用场景
曼哈顿距离h=∥x1−x2∥+∥y1−y2∥h = \|x_1 - x_2\| + \|y_1 - y_2\|h=∥x1​−x2​∥+∥y1​−y2​∥四方向网格移动
对角距离h=max⁡(∥dx∥,∥dy∥)+(2−1)⋅min⁡(∥dx∥,∥dy∥)h = \max(\|dx\|, \|dy\|) + (\sqrt{2} - 1) \cdot \min(\|dx\|, \|dy\|)h=max(∥dx∥,∥dy∥)+(2​−1)⋅min(∥dx∥,∥dy∥)八方向网格移动
欧几里得距离h=(x1−x2)2+(y1−y2)2h = \sqrt{(x_1 - x_2)^2 + (y_1 - y_2)^2}h=(x1​−x2​)2+(y1​−y2​)2​任意方向移动

4.3 算法流程

function A_Star(graph, start, goal, heuristic):
    g ← {start: 0}
    f ← {start: heuristic(start, goal)}
    open_set ← MinHeap([(f[start], start)])
    parent ← {start: null}
    closed_set ← {}

    while open_set is not empty:
        (_, current) ← open_set.extract_min()
        if current == goal:
            return reconstruct_path(parent, goal)
        closed_set.add(current)

        for (neighbor, weight) in graph.neighbors(current):
            if neighbor in closed_set:
                continue
            tentative_g ← g[current] + weight
            if tentative_g < g.get(neighbor, ∞):
                g[neighbor] ← tentative_g
                f[neighbor] ← tentative_g + heuristic(neighbor, goal)
                parent[neighbor] ← current
                open_set.insert((f[neighbor], neighbor))

    return no path found

4.4 性能与特性

指标值
时间复杂度依赖于启发函数质量，最优情况 O(d)O(d)O(d)，最坏退化为 Dijkstra
空间复杂度O(V)O(V)O(V)（Open + Closed 集合）
最优性可采纳启发函数下保证最短路径

4.5 启发函数质量的影响

启发函数的精度直接决定 A* 的搜索效率：

  • h(n)=0h(n) = 0h(n)=0：退化为 Dijkstra，失去方向引导。
  • h(n)=h∗(n)h(n) = h^*(n)h(n)=h∗(n)：完美启发函数，A* 直接沿最短路径搜索，不探索任何多余节点。
  • h(n)>h∗(n)h(n) > h^*(n)h(n)>h∗(n)：不可采纳，可能找到非最优路径，但搜索速度更快（Weighted A*）。

5. 双向搜索（Bidirectional Search）

5.1 原理

双向搜索同时从起点和终点出发进行搜索，当两个搜索前沿相遇时，拼接两段路径构成完整路径。

对于 BFS 而言，搜索空间可近似为以起点/终点为圆心的圆。单向 BFS 搜索面积为 πr2\pi r^2πr2，双向 BFS 的搜索面积为 2π(r/2)2=πr2/22 \pi (r/2)^2 = \pi r^2 / 22π(r/2)2=πr2/2，理论上将搜索空间减半。

5.2 双向 Dijkstra

双向 Dijkstra 维护两个优先队列，交替扩展 ggg 值最小的节点。终止条件为：当某个节点同时被两个搜索前沿处理过，且当前最优路径的代价不大于两个队列中最小的 ggg 值之和时，搜索终止。

5.3 双向 A*

双向 A* 的启发函数设计更为复杂，需要保证一致性（Consistency）。一种常用方法是使用对称启发函数：

  • 正向搜索的启发函数：hf(n)=h(n,t)−h(n,s)+h(s,t)2h_f(n) = \frac{h(n, t) - h(n, s) + h(s, t)}{2}hf​(n)=2h(n,t)−h(n,s)+h(s,t)​
  • 反向搜索的启发函数：hb(n)=h(n,s)−h(n,t)+h(s,t)2h_b(n) = \frac{h(n, s) - h(n, t) + h(s, t)}{2}hb​(n)=2h(n,s)−h(n,t)+h(s,t)​

5.4 适用场景

双向搜索在大规模图上效果显著，尤其适用于道路网络等稀疏图。但在狭窄通道或迷宫结构中，两个搜索前沿可能无法有效相遇，退化为接近单向搜索的性能。

6. 跳点搜索（Jump Point Search, JPS）

6.1 原理

JPS 是 A* 在均匀权重网格图上的优化变体，由 Harabor 和 Grastien 于 2011 年提出。其核心思想是：在均匀网格中，很多对称路径具有相同的代价，A* 会将这些等价路径的中间节点全部加入 Open 集合，造成大量冗余。

JPS 通过定义”跳点”（Jump Point）来消除这种对称性。跳点是路径中必须转向的关键节点，搜索时直接在直线方向上”跳跃”到下一个跳点，跳过所有中间节点。

6.2 跳点的定义

一个节点是跳点，当且仅当：

  1. 它是终点。
  2. 它至少有一个强制邻居（Forced Neighbor）——即一个因障碍物存在而只能通过当前节点到达的邻居。

6.3 性能特性

指标值
时间复杂度理论上与 A* 相同，但实际扩展节点数大幅减少
空间复杂度O(V)O(V)O(V)（但 Open 集合显著更小）
最优性保证最短路径
限制仅适用于均匀权重网格图

JPS 通常能将 A* 的扩展节点数减少一个数量级以上，在开阔地图中效果尤为显著。

7. 导航网格（Navigation Mesh）

7.1 原理

导航网格是 3D 游戏引擎中广泛使用的寻路方案。其基本思想是将可行走区域分解为一组凸多边形（通常是三角形），每个多边形内部的任意两点之间都存在无障碍直线路径。

寻路过程分为两步：

  1. 高层路径搜索：将多边形作为图节点，多边形之间的共享边作为图边，使用 A* 搜索从起点所在多边形到终点所在多边形的路径序列。
  2. 路径平滑：使用漏斗算法（Funnel Algorithm, Simple Stupid Funnel Algorithm）将多边形序列转化为沿共享边端点的平滑路径。

7.2 漏斗算法

漏斗算法的目标是在多边形通道（Channel）中找到最短路径。它维护一个从当前位置出发的”漏斗”，由左右两条边界线构成，逐步收窄直到到达终点。

7.3 优势与权衡

优势权衡
高层图节点数远少于网格节点数，搜索极快需要预处理生成导航网格
天然支持不同大小的 Agent（通过 NavMesh 缩放）动态障碍物需要实时更新网格
路径天然平滑，无需后处理阶梯感导航网格生成算法（如 Recast）实现复杂

8. 预处理加速方法

对于超大规模图（如国家级道路网络，节点数 10710^7107+），在线搜索即使使用 A* 也可能不够快。以下预处理方法可将查询时间压缩至亚毫秒级：

8.1 分层方法：Contraction Hierarchies (CH)

CH 通过迭代”收缩”不重要的节点（添加快捷边保持最短路径），构建一个层次图。查询时使用双向 Dijkstra 仅在层次图中向上搜索，极大地减少搜索空间。

  • 预处理时间：分钟级
  • 查询时间：微秒级
  • 空间开销：原图的 2∼32\sim32∼3 倍

8.2 标签方法：Hub Labeling

为每个节点预计算一组”枢纽”（Hub）及到枢纽的距离。查询时仅需比较两个节点的标签集合，取交集中距离之和最小的枢纽。

  • 预处理时间：小时级
  • 查询时间：纳秒至微秒级
  • 空间开销：较大（每节点需存储标签列表）

9. 算法对比与选型指南

算法最优性边权要求适用规模工程复杂度典型应用
BFS等权最优无权/等权中小规模低简单网格游戏
Dijkstra最优非负权重中规模低通用加权图
A*最优非负权重中大规模中游戏、机器人
JPS最优均匀网格大规模网格中2D 游戏
双向搜索最优非负权重大规模中高道路网络
NavMesh + A*近似最优N/A3D 场景高3D 游戏引擎
CH最优非负权重超大规模高地图导航

选型建议

  • 小规模等权网格（如教学或简单游戏）：BFS 足矣。
  • 加权图的通用单源最短路：Dijkstra。
  • 有明确终点的单对最短路：A*，选择合适的启发函数。
  • 大规模均匀网格（如 RTS 游戏）：JPS。
  • 3D 游戏场景：NavMesh + A* + 漏斗算法。
  • 超大规模静态图（如全国道路网络）：CH 或 Hub Labeling。
  • 动态图（权重频繁变化）：D* Lite 或增量式 A*。

10. 实现优化要点

10.1 优先队列选型

A* 和 Dijkstra 的性能瓶颈往往在优先队列操作上。常见选择：

数据结构InsertExtract-MinDecrease-Key
二叉堆O(log⁡n)O(\log n)O(logn)O(log⁡n)O(\log n)O(logn)O(log⁡n)O(\log n)O(logn)
斐波那契堆O(1)O(1)O(1) 摊销O(log⁡n)O(\log n)O(logn) 摊销O(1)O(1)O(1) 摊销
桶队列O(1)O(1)O(1)O(C)O(C)O(C)（CCC 为最大权重）O(1)O(1)O(1)

实践中，二叉堆因常数因子小和缓存友好性，通常优于理论上更优的斐波那契堆。对于整数权重的网格图，桶队列（Bucket Queue）是最快的选择。

10.2 Closed 集合的表示

对于网格图，使用二维布尔数组替代哈希集合（HashSet）可显著减少缓存未命中：

# 哈希集合方式（通用但较慢）
closed = set()
closed.add((x, y))

# 数组方式（网格图专用，更快）
closed = [[False] * cols for _ in range(rows)]
closed[x][y] = True

10.3 路径平滑

网格图上 A* 产生的路径通常有明显的”阶梯感”。后处理方法包括：

  • 射线投射（Line-of-Sight Check）：依次检查路径中相隔的节点是否可以直连，移除中间冗余节点。
  • Catmull-Rom 样条：对路径关键点进行曲线插值，生成平滑运动轨迹。
  • Theta*：A* 的变体，在搜索过程中直接进行视线检查，产出天然平滑的路径。

总结

寻路算法的选型没有银弹。BFS 和 Dijkstra 提供了坚实的理论基础，A* 通过启发函数引入了方向性，JPS 和 NavMesh 分别针对网格和 3D 场景进行了专门优化，而 CH 等预处理方法则将查询时间推至极限。理解每种算法的假设条件与性能边界，是在工程实践中做出正确选型的关键。
