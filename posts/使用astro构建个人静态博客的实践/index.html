使用 Astro 构建个人静态博客的实践

为什么选择 Astro

在搭建个人博客时，可选的技术方案众多：从传统的 WordPress、Hexo，到现代的 Next.js、Nuxt，再到以内容为中心的 Gatsby。最终选择 Astro 的原因主要有以下几点：

1. 默认零 JS 输出

Astro 的核心理念是”零运行时 JavaScript”——除非开发者主动使用 client:* 指令，否则组件不会向浏览器输出任何 JavaScript。对于内容为主的博客站点来说，这意味着极快的首屏加载速度和极高的 Lighthouse 评分。

2. Content Collections 内容集合

Astro 提供了内置的 Content Collections 模块，可以通过 Zod Schema 对 Markdown/MDX 文章的 frontmatter 进行类型校验，避免数据字段不一致的问题，同时提供完整的 TypeScript 类型推断。

3. 框架无关性

Astro 支持在同一个项目中混用 React、Vue、Svelte 等多种 UI 框架的组件。即便后续需要引入某个框架生态中的特定组件，也不需要全量迁移。

4. 静态优先，易于部署

输出为纯静态文件，可以直接部署到 GitHub Pages、Cloudflare Pages、Vercel 等任意静态托管平台，无需维护服务器。

项目结构

一个典型的 Astro 博客项目结构如下：

src/
├── content/
│   ├── config.ts          # Content Collections schema 定义
│   └── posts/             # Markdown/MDX 文章存放目录
│       └── article.md
├── layouts/
│   └── Layout.astro       # 全局布局（head、导航、footer）
├── pages/
│   ├── index.astro        # 首页
│   ├── posts/
│   │   ├── index.astro    # 文章列表页
│   │   └── [...slug].astro  # 文章动态路由页
│   └── rss.xml.ts         # RSS 订阅源
├── components/            # 可复用组件
├── data/                  # 静态数据文件（JSON/YAML）
└── styles/                # 全局样式

Content Collections 的使用

在 src/content/config.ts 中定义文章的字段 Schema：

import { defineCollection, z } from 'astro:content';

const postsCollection = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    date: z.union([z.string(), z.date()]).transform((val) => new Date(val)),
    tags: z.array(z.string()).optional().default([]),
    categories: z.array(z.string()).optional().default([]),
    description: z.string().optional(),
    cover: z.string().optional(),
    draft: z.boolean().optional().default(false),
  }),
});

export const collections = { posts: postsCollection };

在页面中获取文章列表：

---
import { getCollection } from 'astro:content';

const posts = await getCollection('posts', ({ data }) => !data.draft);
const sorted = posts.sort((a, b) => b.data.date.valueOf() - a.data.date.valueOf());
---

在动态路由页中渲染单篇文章：

---
// pages/posts/[...slug].astro
import { getCollection, render } from 'astro:content';

export async function getStaticPaths() {
  const posts = await getCollection('posts');
  return posts.map(post => ({
    params: { slug: post.slug },
    props: { post },
  }));
}

const { post } = Astro.props;
const { Content } = await render(post);
---

<article>
  <h1>{post.data.title}</h1>
  <Content />
</article>

视图切换动画

Astro 内置了 <ViewTransitions> 组件，可以在不同页面之间实现平滑的过渡动画，并且兼容浏览器原生的 View Transitions API：

---
import { ViewTransitions } from 'astro:transitions';
---

<head>
  <ViewTransitions />
</head>

启用后，页面间导航会呈现淡入淡出效果，无需额外配置。如需自定义特定元素的过渡，可以使用 transition:name 指令标记共享元素：

<img src={cover} transition:name={`cover-${post.slug}`} />

静态搜索实现

Astro 不提供服务端运行时，无法直接接入 Elasticsearch 等搜索服务，但可以在构建阶段生成静态搜索索引。

在构建时输出 JSON 格式的文章摘要索引文件，然后在客户端通过简单的字符串匹配实现即时搜索：

// 构建时：在 rss.xml.ts 或专用 API 端点中导出
const searchIndex = posts.map(p => ({
  title: p.data.title,
  slug: p.slug,
  tags: p.data.tags,
  excerpt: p.body.slice(0, 200),
}));

客户端：

const query = input.value.toLowerCase();
const results = searchIndex.filter(item =>
  item.title.toLowerCase().includes(query) ||
  item.tags.some(t => t.toLowerCase().includes(query))
);

这种方案完全无需后端，对于中小规模的博客（数百篇文章以内）性能表现良好。

部署到 GitHub Pages

astro.config.mjs 配置：

import { defineConfig } from 'astro/config';

export default defineConfig({
  site: 'https://username.github.io',
  base: '/repo-name', // 如果部署在子路径下
  output: 'static',
});

GitHub Actions 工作流（.github/workflows/deploy.yml）：

name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist
      - uses: actions/deploy-pages@v4

小结

Astro 作为博客站点框架的优势总结：

维度表现
页面性能默认零 JS，首屏极快
内容管理Content Collections 类型安全
扩展性支持多框架混用
部署成本纯静态，任意 CDN 托管
学习曲线组件语法接近 HTML，上手快

对于以文字内容为核心的个人博客场景，Astro 是目前综合体验较好的选择之一。
