Franka Emika 控制架构解析：从 FCI 到 Polymetis 的软件栈

Franka Emika Panda 是具身智能研究中使用最广泛的协作机械臂之一。其开放的控制接口（FCI）允许研究者以 1 kHz 频率进行底层力矩控制。然而，从 FCI 到可用的策略执行环境之间，存在多层软件抽象。

本文从底层 FCI 出发，逐层解析 Franka 的控制软件栈。

1. 硬件层

1.1 Franka Emika Panda

规格值
自由度7 (冗余)
有效负载3 kg
工作半径855 mm
关节力矩传感器7 个（每关节一个）
通信接口1 Gbps 以太网 (FCI)
内部控制频率1 kHz

1.2 实时以太网通信

FCI 使用 UDP 协议在 1 kHz 频率下与控制工作站双向通信。每个控制周期（1 ms）内，工作站必须完成：

接收状态→计算控制指令→发送指令\text{接收状态} \rightarrow \text{计算控制指令} \rightarrow \text{发送指令}接收状态→计算控制指令→发送指令

超时将触发通信违规错误（Communication Violation），导致机械臂急停。

2. libfranka：底层 C++ 客户端库

2.1 职责

libfranka 是 Franka 官方提供的 C++ 客户端库，负责：

  • 建立与机械臂的实时 UDP 连接。
  • 以 1 kHz 频率接收关节状态（位置、速度、力矩、外部力矩等）。
  • 以 1 kHz 频率发送控制指令。
  • 实现基本的安全检查（力矩限位、速度限位、加速度限位）。

2.2 控制模式

libfranka 支持多种控制模式：

模式接口控制频率
关节力矩控制control(torque_callback)1 kHz
关节位置控制control(position_callback)1 kHz
关节速度控制control(velocity_callback)1 kHz
笛卡尔位置控制control(cartesian_pose_callback)1 kHz
笛卡尔速度控制control(cartesian_velocity_callback)1 kHz

所有模式的回调函数必须在 1 ms 内返回。

2.3 实时性要求

libfranka 的回调运行在实时线程中。为保证 1 kHz 的确定性执行：

  • 控制工作站应使用 PREEMPT_RT 实时内核。
  • 回调函数中不能进行动态内存分配、磁盘 I/O 或网络请求。
  • 工作站到机械臂的网络延迟必须小于 1 ms。

3. franka_ros / franka_ros2：ROS 封装

3.1 职责

franka_ros 将 libfranka 封装为 ROS 节点，提供标准 ROS 接口：

  • 发布关节状态到 /joint_states 话题。
  • 通过 ros_control 框架提供控制接口。
  • 支持 MoveIt 运动规划。

3.2 局限

ROS 的通信机制（Topic/Service）引入了额外延迟和非确定性。虽然 franka_ros 的控制循环本身运行在实时线程中，但与 ROS 生态系统交互的部分不具有实时保证。

4. Polymetis：Facebook Research 的中间件

4.1 设计动机

Polymetis（Facebook Research）是专门为机器人学习研究设计的控制中间件。其设计目标是：

  • 提供高层 Python API，降低控制代码的编写门槛。
  • 通过 gRPC 实现控制工作站与策略服务器之间的网络透明通信。
  • 保持底层 1 kHz 实时控制的能力。

4.2 架构

┌─────────────────────────────────────────┐
│  策略服务器（Python，可在 GPU 机器上）     │
│  → 策略推理 + 动作生成                    │
│  → gRPC 客户端                           │
└───────────────┬─────────────────────────┘
                ↓ gRPC (网络)
┌─────────────────────────────────────────┐
│  Polymetis 控制服务器（C++，NUC 上）      │
│  → gRPC 服务端                           │
│  → 控制回路（1 kHz）                     │
│  → libfranka 接口                        │
└───────────────┬─────────────────────────┘
                ↓ UDP (以太网)
┌─────────────────────────────────────────┐
│  Franka Emika Panda                     │
└─────────────────────────────────────────┘

4.4 控制流程

from polymetis import RobotInterface

robot = RobotInterface(ip_address="10.10.10.1")

# 读取当前关节位置
joint_pos = robot.get_joint_positions()

# 发送关节位置指令
robot.set_joint_positions(target_joint_pos)

# 发送笛卡尔阻抗控制指令

robot.update_desired_ee_pose(target_pose)

5. 上层策略集成

5.1 DROID 框架

DROID 在 Polymetis 之上构建了数据采集和策略执行框架：

DROID Server (NUC) → Polymetis → libfranka → Franka
DROID Laptop → 相机采集 + 策略推理 → DROID Server

5.2 OpenPI 集成

OpenPI 的 VLA 策略服务器通过 WebSocket 向 DROID 客户端输出动作：

OpenPI Policy Server (GPU) → WebSocket → DROID Adapter → Polymetis → Franka

5.3 频率分层

完整系统的频率分层：

层频率组件
VLA 推理2-10 HzOpenPI 服务器
策略执行10-50 HzDROID 客户端
轨迹插值1 kHzPolymetis 控制器
底层伺服1 kHzlibfranka → Franka

6. 常见问题

6.1 通信违规

现象：communication_constraints_violation，机械臂急停。

原因：控制回调未在 1 ms 内完成。常见原因包括控制工作站未使用 RT 内核、网络延迟过高、回调中执行了阻塞操作。

6.2 反射运动违规

现象：joint_motion_generator_velocity_limits_violation

原因：指令的关节速度变化超过限值。通常由于目标位置跳变过大或插值不够平滑。

6.3 gRPC 超时

现象：Polymetis Python 客户端连接超时。

原因：网络配置问题，检查 IP 地址、端口和防火墙设置。

总结

Franka 的控制软件栈从底向上为：FCI 硬件接口 → libfranka C++ 库 → Polymetis 中间件 → 上层策略框架（DROID/OpenPI）。每一层都在实时性与易用性之间做出权衡：libfranka 提供最低延迟但使用门槛高，Polymetis 提供 Python API 和网络透明性但引入 gRPC 延迟，上层框架提供完整的数据采集和策略执行流程但频率受限于推理速度。理解各层的职责和约束，是将学习策略成功部署到 Franka 机械臂上的基础。
