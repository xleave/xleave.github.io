长时域任务规划：LLM 作为机器人高级规划器的可行性分析

#LLM#任务规划#具身智能#分层控制#长时域

在机器人操作的复杂任务中（如”整理桌面”或”做一杯咖啡”），系统需要将高层目标分解为有序的子任务序列，再将每个子任务转化为底层动作。大语言模型（LLM）凭借其丰富的世界知识和推理能力，被越来越多地用作机器人的高级任务规划器。

本文分析 LLM 在长时域任务规划中的应用模式、接地问题和当前局限。

1. 长时域任务的挑战

1.1 任务分解

高层指令”make a cup of coffee”需要分解为：

1. 走到咖啡机前
2. 拿起咖啡杯
3. 将杯子放到咖啡机下方
4. 按下启动按钮
5. 等待咖啡制作完成
6. 取回杯子

这种分解需要关于咖啡制作流程的常识知识，以及对物理世界的因果理解（如”必须先放杯子再按按钮”）。

1.2 为什么端到端模型难以处理？

端到端 VLA 模型直接从观测映射到动作，理论上可以处理任意长的任务。但实践中存在以下困难：

  • 信用分配：数百步的轨迹中，难以将最终成功/失败归因到具体动作。
  • 组合爆炸：子任务的排列组合空间随任务长度指数增长。
  • 数据需求：长时域任务的完整演示数据收集成本极高。

2. LLM 作为规划器的基本架构

2.1 分层架构

┌──────────────────────────────────────────┐
│  LLM 高级规划器                            │
│  输入: 高层指令 + 场景描述                   │
│  输出: 子任务序列 [task_1, task_2, ...]     │
└─────────────────┬────────────────────────┘
                  ↓
┌──────────────────────────────────────────┐
│  低层策略执行器（VLA / RL Policy）          │
│  输入: 子任务指令 + 观测                    │
│  输出: 底层动作序列                         │
└──────────────────────────────────────────┘

LLM 负责”做什么”（What to do），低层策略负责”怎么做”（How to do it）。

2.2 代表工作

工作LLM 角色接地方式
SayCan (Google, 2022)生成候选子任务通过 affordance 函数过滤不可行的子任务
Code as Policies (2022)生成 Python 代码作为动作计划调用预定义的机器人 API
Inner Monologue (2022)生成子任务 + 监控执行通过感知反馈闭环修正
VoxPoser (2023)生成 3D value map 引导操作LLM 生成代码，代码操纵 3D 场
π0.5 (2024)VLA 内部的 CoT 推理模型内部生成子目标后直接执行

3. 接地问题（Grounding）

3.1 什么是接地？

LLM 拥有丰富的语言知识（“杯子可以用来装水”），但缺乏与具体物理环境的绑定。接地（Grounding）是指将 LLM 的抽象知识与当前机器人的具体场景连接：

  • 物体接地：LLM 说”拿起杯子”，但场景中有哪些杯子？在哪里？
  • 能力接地：LLM 计划”打开抽屉”，但当前机器人是否具备打开抽屉的技能？
  • 状态接地：LLM 计划”倒水”，但杯子里当前有水吗？

3.2 接地方法

方法一：Affordance Scoring（SayCan）

对于 LLM 生成的每个候选子任务，使用一个预训练的 affordance 函数评估该任务在当前场景中的可行性：

score(task)=PLLM(task∣goal)×Paffordance(task∣scene)\text{score}(task) = P_{\text{LLM}}(task | \text{goal}) \times P_{\text{affordance}}(task | \text{scene})score(task)=PLLM​(task∣goal)×Paffordance​(task∣scene)

LLM 提供”这个任务与目标相关”的概率，affordance 函数提供”这个任务在当前场景中可执行”的概率。两者相乘，选择综合得分最高的子任务。

方法二：场景描述注入

将当前场景的结构化描述作为 LLM 的 prompt 上下文：

prompt = f"""
You are a robot assistant. The scene contains:
- A red cup at position (0.3, 0.5)
- A blue plate at position (0.1, 0.2)
- A coffee machine on the left counter

The robot can perform: pick, place, push, press_button

Task: Make a cup of coffee.
Generate a step-by-step plan using only available actions.
"""
plan = llm.generate(prompt)

方法三：代码生成（Code as Policies）

让 LLM 生成可执行的 Python 代码，代码通过预定义的 API 与机器人交互：

# LLM 生成的代码
cup = detect("cup")
pick(cup)
move_to(coffee_machine.slot)
place(cup)
press(coffee_machine.button)
wait(30)  # 等待 30 秒
pick(cup)

这种方式的接地通过 API 的类型系统和运行时错误检查隐式实现。

4. 闭环规划

4.1 开环 vs. 闭环

开环规划：LLM 在任务开始时一次性生成完整计划，然后按顺序执行，不根据中间结果调整。

问题：执行过程中可能出现预期外的情况（物体掉落、抓取失败、场景变化）。

闭环规划：LLM 在每个子任务执行后获取执行结果的反馈，动态调整后续计划。

4.2 反馈来源

反馈类型来源信息内容
成功/失败信号低层策略的执行结果当前子任务是否成功完成
场景变化描述视觉感知系统执行后场景发生了什么变化
异常报告安全监控系统是否触发安全约束
人类指令用户交互界面用户的修正或补充指令

4.3 Inner Monologue 模式

Inner Monologue（2022）提出的闭环模式：

LLM: 计划子任务 → "pick up the cup"
Robot: 执行 → 失败（杯子太滑）
感知系统: 反馈 → "the cup is still on the table"
LLM: 重新规划 → "use both hands to pick up the cup"
Robot: 执行 → 成功
LLM: 下一步 → "place the cup under the coffee machine"
...

5. LLM 规划器的局限

5.1 物理推理能力不足

LLM 对物理世界的理解是基于语言语料的统计相关性，而非真正的物理模拟。

示例：

  • LLM 可能计划”将杯子倒扣在桌上放入水”——违反重力常识。
  • LLM 可能不理解”推一个重物需要更大的力”。

5.2 空间推理的局限

LLM 在处理精确的空间关系时表现较弱：

  • “将 A 放到 B 的左边 10 厘米” → LLM 对”10 厘米”的物理含义理解模糊。
  • 涉及三维旋转或姿态的空间推理更加困难。

5.3 延迟与成本

每次调用 LLM 引入数百毫秒到数秒的延迟。对于需要快速重规划的动态环境，LLM 的响应速度可能不足。

5.4 幻觉

LLM 可能生成听起来合理但实际不可行的计划：

LLM: "使用微波炉加热杯子"
（但场景中没有微波炉）

接地机制（如 affordance scoring）可以部分缓解这一问题，但无法完全消除。

6. 与端到端 VLA 的融合趋势

π0.5 代表了 LLM 规划与低层策略融合的新趋势：不再使用外部 LLM 作为独立规划器，而是在 VLA 模型内部引入高级语言推理（Chain-of-Thought），模型先自回归生成子目标描述，然后基于子目标生成动作。

这种融合的优势：

  • 端到端优化：规划与执行共享特征表示，可联合训练。
  • 低延迟：无需外部 LLM API 调用。
  • 隐式接地：模型的视觉编码器直接提供场景信息，无需显式场景描述构建。

总结

LLM 作为机器人高级规划器，利用其丰富的世界知识将高层目标分解为可执行的子任务序列。接地问题——将抽象知识与具体环境绑定——是该范式的核心挑战，当前通过 affordance scoring、场景描述注入和代码生成等方法部分解决。闭环规划通过执行反馈提升了系统的鲁棒性。长期来看，LLM 规划能力与低层策略的端到端融合（如 π0.5 的模型内 CoT）可能成为主流方向。
