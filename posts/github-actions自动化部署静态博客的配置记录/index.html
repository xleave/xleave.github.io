GitHub Actions 自动化部署静态博客的配置记录

为什么需要 CI/CD

静态博客的部署流程本质上很简单：npm run build → 将 dist/ 目录推送到目标平台。但手动执行这些步骤存在几个问题：

  • 每次写完文章需要手动触发构建，容易忘记；
  • 本地构建环境可能与最终部署环境存在差异；
  • 无法记录构建历史，难以排查部署异常；
  • 多人协作时，各自构建容易产生不一致的输出。

使用 GitHub Actions 后，只需将代码推送到指定分支，构建与部署流程就会自动执行，整个过程有完整的日志记录。

GitHub Pages 部署配置

仓库设置

首先在仓库的 Settings > Pages 中将 Source 设置为 GitHub Actions，而非传统的 gh-pages 分支方式。这是 GitHub 目前推荐的方式，可以使用更灵活的部署权限控制。

Workflow 文件

在项目根目录创建 .github/workflows/deploy.yml：

name: Build and Deploy

on:
  push:
    branches:
      - main
  workflow_dispatch:  # 允许手动触发

# 设置 GITHUB_TOKEN 的权限
permissions:
  contents: read
  pages: write
  id-token: write

# 只允许一个并发部署，新推送会取消正在运行的旧部署
concurrency:
  group: "pages"
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'  # 自动缓存 node_modules

      - name: Install dependencies
        run: npm ci  # 使用 ci 而非 install，确保使用 lock 文件

      - name: Build site
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

关键配置说明

npm ci vs npm install

npm ci 会严格按照 package-lock.json 安装依赖，不会升级任何包。相比 npm install，它：

  • 保证构建的可重现性
  • 更快（跳过依赖解析步骤）
  • 如果 package-lock.json 与 package.json 不一致会直接报错

cache: 'npm'

actions/setup-node 的 cache 选项会自动缓存 npm 的包缓存目录，在依赖未变化的情况下，后续构建可以跳过网络下载步骤，显著减少构建时间。

Astro 配置

astro.config.mjs 中需要配置正确的站点地址，否则内部链接和资源路径会出错：

import { defineConfig } from 'astro/config';

export default defineConfig({
  site: 'https://your-username.github.io',
  // 如果部署在子路径（如 /repo-name/），取消注释：
  // base: '/repo-name',
  output: 'static',
});

对于部署在自定义域名（如 yourdomain.com）下的站点，base 通常不需要设置。

依赖更新自动化（Dependabot）

GitHub 提供了 Dependabot 自动创建依赖更新 PR 的功能。在 .github/dependabot.yml 中配置：

version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 5
    # 自动合并 patch 版本更新（如安全修复）
    groups:
      patch-updates:
        update-types:
          - "patch"

对于 patch 级别的更新（通常是安全修复或 bug 修复），可以在确认 CI 通过后直接合并，不需要详细审查。对于 minor 和 major 级别的更新，建议手动审阅 changelog 后再决定是否合并。

构建失败的常见原因

1. 类型错误（TypeScript/Astro 类型检查）

Astro 在构建时会对 .astro 文件进行类型检查。如果 frontmatter 中的字段与 config.ts 中定义的 Schema 不匹配，构建会失败。

例如：在 config.ts 中 date 字段定义为必填，但某篇文章的 frontmatter 中缺少 date 字段，就会触发错误。

2. 图片路径不存在

构建时，public/ 目录之外的图片引用（如使用 src 属性的 <Image /> 组件）会被 Astro 验证文件是否存在。路径错误会导致构建失败。

3. 环境变量缺失

如果项目使用了 import.meta.env.YOUR_KEY 形式的环境变量，需要在 GitHub Actions 的 Settings > Secrets and variables > Actions 中预先配置这些变量，并在 workflow 中通过 env: 注入：

- name: Build site
  run: npm run build
  env:
    SOME_API_KEY: ${{ secrets.SOME_API_KEY }}

手动触发与分支策略

Workflow 中的 workflow_dispatch 触发器允许在 GitHub Actions 页面手动触发构建，适用于以下场景：

  • 部署配置修改但代码未变动；
  • 需要紧急重新部署；
  • 调试 workflow 本身。

对于写作内容较多的博客，可以考虑使用 draft: true frontmatter 字段配合 workflow 条件判断，将草稿排除在构建之外，避免未完成的文章被意外发布：

// 在获取文章集合时过滤草稿
const posts = await getCollection('posts', ({ data }) => !data.draft);

小结

使用 GitHub Actions 部署静态博客的核心流程：

代码推送到 main 分支
    ↓
Actions: checkout → setup-node → npm ci → npm run build
    ↓
上传 dist/ 为 Pages artifact
    ↓
deploy-pages 将 artifact 发布到 GitHub Pages CDN
    ↓
站点更新完成（通常 1-3 分钟内）

这套方案完全免费（GitHub Actions 免费额度对静态博客而言绰绰有余），且无需维护任何服务器基础设施。
